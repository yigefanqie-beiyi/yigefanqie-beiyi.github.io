题目分析：

题目提供了一个测试程序，大致功能是：通过创建多个进程，在多个CPU上运行重复的获取、释放内存的操作，但是由于XV6原始内存分配的策略是，全局共享一个内存池，通过结构体kmem进行管理，因此多个进程在并行执行sbrk操作时，会出现竞争，这就需要锁的使用来保证不会出现错误，但多个进程都用一个锁，意味着并行退化成了串行，系统效率会大大降低。

我们需要做的是：修改分配内存的逻辑，给每个cpu都创建一个空闲内存链表，也就是将原本的内存分配结构管理体kmem修改成数组kmems[ NCPU ]，尽量避免太多的竞争，因此需要修改kmalloc.c里面的一系列函数，通过提供的NCPU可以知道当前cpu核心的数量，通过函数cpuid()可以获得当前所在的cpu的id（使用这个函数要记得开关中断）

当一个cpu上对应的空闲内存不足，但仍需分配内存时，需要从别的cpu的空闲内存链表中获取

Buffer cache (hard)

简而言之，就是针对文件系统里的buffer cache层进行优化，原设计是，buffer cache里通过一个双向链表，通过LRU策略和一个全局的bcache结构体进行管理，因此也会出现并行访问文件系统时，对bcache的锁的竞争，为了减小这种竞争，我们进行优化

这里有一些提示：

  • 阅读 xv6 书籍中的 块缓存（block cache） 相关章节（第 8.1-8.3 章）。

  • 可以使用固定数量的哈希桶，不必动态调整哈希表大小。使用质数作为桶数量（例如 13），以减少哈希冲突的可能性。

  • 在哈希表中搜索缓冲块（buffer）并分配新块时，必须是原子操作。

  • 移除所有缓冲区链表（如 bcache.head 等），不要实现 LRU（最近最少使用）。 这样 brelse() 就不需要获取 bcache.lock 了。在 bget() 中，你可以选择任何 refcnt == 0 的块，而不是最少使用的块。

  • 你可能无法同时检查缓存是否命中并在缓存未命中时找到可替换的块；因此，可能需要释放所有锁并重新查找。

  • 在某些情况下，你的解决方案可能需要同时持有两个锁。例如，在块淘汰（eviction）过程中，可能需要同时持有 bcache.lock 和某个桶的锁。确保避免死锁。

  • 当替换一个块时，该块可能会从一个桶移动到另一个桶（因为新块的哈希值可能不同）。如果新旧块的哈希桶相同，可能会有死锁风险，需要特别注意。

一些调试技巧：

先实现桶锁，但仍保留全局 bcache.lock（在 bget() 开头和结尾），以串行化代码，确保正确性。

在确认没有竞态问题后，移除全局锁，处理并发性问题。

使用 make CPUS=1 qemu 在单核模式下运行，方便调试。

使用 xv6 的竞态检测器查找潜在的竞争条件（使用 make KCSAN=1 qemu 运行）。

思路：

  1. 按照给的hints，直接固定数量哈希桶为13，哈希函数为对13取模，将文件系统的每个blockno映射到哈希桶里
  2. 每个哈希桶都设置各自的cache锁，因此将原本只对bcache.lock的竞争变为了对13个哈希桶中某个桶的锁竞争，以减小竞争概率
  3. 当要读取一个block时，按照以下顺序判断：
    • 先根据HASH在自己的桶里查找
    • 没找到，则在bcache.buf里查找有无空闲buf，添加到自己的哈希桶中
    • 如果没有空闲buf，则在别的桶里查找有无refcnt为0的buf，并转移到自己的桶里
  4. 其实没搞懂这个实验的意义，应该就是让我们体验一下lock的用处，甚至要求我们删除LRU，不理解

其它函数都是小修小改，主要是bget函数要重写逻辑，并注意锁的使用
