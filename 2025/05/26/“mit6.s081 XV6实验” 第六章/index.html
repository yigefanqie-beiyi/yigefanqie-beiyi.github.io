“mit6.s081 XV6实验” Lab6 Multithreading

前置知识：XV6 Book 第七章源码阅读

swtch.S: void swtch(struct context *old, struct context *new);

swtch.S文件实现了xv6操作系统中的上下文切换函数，这是操作系统中线程或进程切换的核心机制。主要由三个功能：

  1. 保存old进程的上下文context
  2. 恢复new进程的上下文context
  3. 通过ret，将new进程的ra寄存器弹出到PC，从而切换到new进程继续执行上下文

s0到s11寄存器是保存寄存器(Saved Registers)，也称为被调用者保存寄存器(Callee-Saved Registers)。这些寄存器在RISC-V的调用约定(Calling Convention)中扮演着重要角色。当一个函数调用另一个函数时，被调用函数必须保证在返回前恢复这些寄存器的原始值。如果函数需要使用这些寄存器，它必须先保存原始值，然后在返回前恢复。

proc.c:

void sched(void)

这个函数的主要目的是执行从用户进程到调度器的上下文切换，为CPU调度其他进程做准备。

  1. 经过三道安全检查：
    • 检查必须持有进程锁，防止进程状态在切换过程中被修改
    • 检查进程只能持有一个锁，也就是进程锁，如果持有其他锁，接下来该进程会被挂起，锁也就不会被释放，其他进程如果想要获取被持有的锁，就会产生死锁的情况
    • 确保进程状态不是RUNNING，这是整个状态转换流程的一环，进程先将自己的状态改为RUNNABLE、SLEEPING或ZOMBIE才能调用sched
  2. 保存进程的中断状态
  3. 调用swtch函数进行old进程的上下文的保存（返回地址被保存到ra寄存器中），shceduler的上下文恢复，同时跳转到shceduler继续调度下一个进程

void scheduler(void)

void yield(void)

允许一个进程主动放弃CPU使用权，让调度器有机会选择其他进程执行。
进程把自己的状态设置为RUNNABLE，然后调用sched将控制权交回给scheduler

梳理一下上述函数：swtch，sched，scheduler

  1. 在一个cpu初始化后，就会进入scheduler函数，里面是一个无限循环，一直重复for循环检测proc数组中进程的状态是否为RUNNABLE，如果是才可以进入下一步，否则继续轮询下一个进程状态
  2. 如果轮询到进程状态是RUNNABLE，则进入swtch函数完成上下文的保存和切换，swtch(&c->context, &p->context);，从scheduler的上下文切换到进程上下文
  3. 如果进程一直顺利执行直到结束，则会通过swtch的ret返回到scheduler，然后scheduler继续循环找下一个RUNNABLE的进程
  4. 如果进程在执行过程中被中途调度，例如时间片用完，IO等待，或者主动yield，则会调用sched函数进行安全检查，并在最后面调用swtch函数swtch(&p->context, &mycpu()->context);切换回scheduler，让scheduler去选择下一个进程进行调度。

mycpu和myproc

  1. tp寄存器(Thread Pointer)通常用于线程局部存储，在xv6中，系统启动时会为每个CPU核心设置唯一的ID，这个ID被存储在每个CPU的tp寄存器中
  2. cpuid通过内联汇编获得当前代码所在的cpu编号
  3. mycpu通过cpuid获得的cpu编号，在cpu数组里获得对应的cpu结构体指针
  4. myproc通过mycpy获取的cpu结构体指针，通过里面的proc结构体元素，获得当前的进程结构体

具体为什么tp寄存器会一直保存着CPU的唯一ID，以后研究。要从start引导的时候开始看起。

Uthread: switching between threads (moderate)

就是一个做用户线程切换的实验，有三个用户线程，要正常切换，那就模仿内核scheduler写一个类似swtch的函数，其实就是一模一样复制下来，然后要在程序执行的时候将要执行的线程处理函数存储到固定位置。

该固定位置可以直接写在给出的结构体的栈里面，将栈顶指针

主要就是设置两个寄存器，ra寄存器存返回地址，sp寄存器存这个线程的栈顶位置

Using threads (moderate)

这个实验主要是探索：lock对多线程的影响，因此要求要在多核的os下进行实验，不能利用xv6本身进行实验

有一个ph的文件，可以执行，里面大概就是对一个哈希表进行多次重复的put和get，如果是单核的情况下，无论开几个线程都是不会冲突的，因为这种单核的多线程其实是一种“伪并行”，但是在多核的情况下，多线程才能实现一个真正的并行加速效果，但会因为竞争条件而付出代价，例如：

put函数下竞争：

  • 线程T1要插入一个新的E1项，因此要将E1插入到表头E3，也就是E1指向表头E3后返回。
  • 与此同时，线程T2在另一个CPU核要插入一个新的E2项，同样的在一个哈希桶里面，因此也将E2插入表头E3，也就是E2指向表头E3，而这个时候，要么丢失E1要么丢失E2，造成错误

get函数下竞争:

  • 假设一个cpu核上进行了put函数插入了一个新的节点
  • 同时，另一个cpu核上进行get函数，但是访问的就是一个没有经过put更新的链表，因此可能会出现数据错误

综上，我们要加锁
这个实验其实很简单，按照文档提示，直接加上锁的定义，初始化，以及获取锁释放锁的操作就可以了，但临界区的设计需要注意

如果设计一个锁，然后每次在get和put里面都直接整体加锁，也能通过实验，但是我们会发现，多核下的速度没有达到最优，并且甚至可能比单核还要低，就是因为临界区设计的太大了，浪费了很多cpu时间

因此，我们观察代码发现，这个哈希表其实就是一个关于5取模而设计的哈希桶，因此对哈希桶每个桶都初始化一个锁，只有在竞争发生在同一个哈希桶时才会触发锁的机制，以加快速度

Barrier(moderate)

在这个作业中，你将实现一个屏障（barrier）：在应用程序中，所有参与的线程必须等待，直到所有其他线程也到达该点后才能继续执行。你将使用 pthread 条件变量（condition variables），这是一种类似于 xv6 的 sleep 和 wakeup 的顺序协调技术。

看一下barrier.c，会发现实际上就是传入多少数字，就创建多少线程，每个线程执行的函数都一样，然后执行就是一个for循环，每次都会调用barrier

关键点：

  • 要用一个round存储当前的轮次（虽然我没看出有什么用）
  • 要注意先后顺序，需要处理一个线程在其他线程退出屏障之前就进入下一轮的情况。特别是，你正在跨轮次重复使用 bstate.nthread 变量。确保离开屏障并快速进入下一轮的线程不会在前一轮仍在使用 bstate.nthread 时增加其值。
  • 记得加锁
